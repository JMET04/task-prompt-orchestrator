# Professional Workflow Accessibility Localization Inclusion Matrix

Use this matrix when a workflow produces user-facing content, software, documents, media, training, reports, forms, dashboards, prompts, packaging, CAD instructions, support flows, public communications, or operational processes that must work across abilities, languages, locales, cultures, devices, literacy levels, network conditions, and access constraints.

## Accessibility Scope

| Code | Access question | Workflow use |
|---|---|---|
| A11YSCOPE01 | Who might be excluded by this workflow or output? | Identify users with visual, auditory, motor, cognitive, language, literacy, device, or network constraints. |
| A11YSCOPE02 | What accessibility standard applies? | Select WCAG, PDF/UA, Section 508, EN 301 549, platform guidelines, internal policy, or domain standard. |
| A11YSCOPE03 | What user-facing artifact needs accessibility review? | Check app, web page, document, slide, spreadsheet, image, video, audio, form, dashboard, or packaging. |
| A11YSCOPE04 | What interaction must remain accessible? | Cover navigation, search, data entry, upload, checkout, approval, download, help, and feedback flows. |
| A11YSCOPE05 | What content must remain accessible? | Cover instructions, headings, labels, tables, charts, diagrams, captions, error messages, and metadata. |
| A11YSCOPE06 | What accessibility risk level applies? | Classify public, internal, regulated, education, government, health, employment, customer-critical, or low-risk use. |
| A11YSCOPE07 | What assistive technology must be supported? | Consider screen readers, keyboard, switch control, captions, magnification, voice input, and braille displays. |
| A11YSCOPE08 | What accommodation path is needed? | Provide alternate format, human support, extended time, simplified mode, or manual fallback. |
| A11YSCOPE09 | What accessibility evidence is required? | Keep checklist, automated scan, manual test, assistive-tech test, user feedback, or signoff. |
| A11YSCOPE10 | What accessibility scope artifact should be produced? | Produce accessibility brief with users, standards, artifacts, risks, tests, and accommodations. |

## Inclusive Requirements

| Code | Access question | Workflow use |
|---|---|---|
| INCLREQ01 | What inclusive outcome is required? | Define equitable access, comprehension, participation, safety, representation, or service quality. |
| INCLREQ02 | What user groups need explicit consideration? | Include age, disability, language, geography, culture, education, role, device, bandwidth, and context. |
| INCLREQ03 | What bias or exclusion could the workflow introduce? | Check examples, defaults, categories, imagery, tone, data, assumptions, and decision rules. |
| INCLREQ04 | What accessibility or inclusion requirement is non-negotiable? | Mark legal, safety, public-service, employment, education, or core product requirements. |
| INCLREQ05 | What inclusive requirement is aspirational? | Separate nice-to-have enhancements from release-blocking needs. |
| INCLREQ06 | What stakeholder should validate inclusion needs? | Consult users, accessibility expert, localization reviewer, support, legal, HR, educator, or community representative. |
| INCLREQ07 | What conflict exists between inclusion and other constraints? | Balance accessibility, cost, speed, brand, technical limits, localization, and privacy. |
| INCLREQ08 | What inclusive design assumption needs validation? | Test assumptions about literacy, terminology, culture, disability, device, and workflow context. |
| INCLREQ09 | What inclusion acceptance criteria apply? | Define measurable criteria for access, comprehension, completion, error recovery, and support. |
| INCLREQ10 | What inclusive requirements artifact should be kept? | Archive requirement list, stakeholders, acceptance criteria, assumptions, and tradeoffs. |

## Content Accessibility

| Code | Access question | Workflow use |
|---|---|---|
| CONTENTA11Y01 | Are headings and structure meaningful? | Use logical headings, lists, tables, landmarks, reading order, and document outline. |
| CONTENTA11Y02 | Is plain language appropriate? | Match reading level, define terms, avoid unnecessary jargon, and provide examples. |
| CONTENTA11Y03 | Are labels and instructions clear? | Make form labels, button names, field help, units, and required fields explicit. |
| CONTENTA11Y04 | Are errors understandable and recoverable? | Explain what failed, where, why, and how to fix it without blame or ambiguity. |
| CONTENTA11Y05 | Do images have useful alternatives? | Provide alt text, captions, long descriptions, or mark decorative images correctly. |
| CONTENTA11Y06 | Are tables readable and navigable? | Use headers, captions, summaries, simple structure, and avoid merged-cell confusion. |
| CONTENTA11Y07 | Are charts and diagrams understandable without color alone? | Add labels, patterns, text summaries, data tables, and direct annotations. |
| CONTENTA11Y08 | Are documents accessible after export? | Check PDF tags, reading order, links, language, title, bookmarks, and selectable text. |
| CONTENTA11Y09 | What content accessibility QA is required? | Run structure review, screen reader spot check, contrast check, and manual content review. |
| CONTENTA11Y10 | What content accessibility artifact should be preserved? | Keep checklist, fixed issues, alt text list, table notes, and export validation. |

## Interaction Accessibility

| Code | Access question | Workflow use |
|---|---|---|
| INTERACT01 | Can the workflow be completed by keyboard alone? | Test focus order, visible focus, shortcuts, dialogs, menus, forms, and escape paths. |
| INTERACT02 | Can screen reader users understand state and controls? | Check names, roles, values, landmarks, live regions, errors, and dynamic updates. |
| INTERACT03 | Are touch and pointer targets usable? | Check size, spacing, gestures, drag/drop alternatives, and accidental activation. |
| INTERACT04 | Are time limits avoidable or adjustable? | Provide pause, extend, save, resume, or no-timeout options where possible. |
| INTERACT05 | Are motion and flashing safe? | Avoid seizure risk, reduce motion, provide controls, and respect system preferences. |
| INTERACT06 | Are complex workflows recoverable? | Allow undo, back, draft save, confirmation, review screen, and support handoff. |
| INTERACT07 | Are authentication and verification steps accessible? | Provide accessible CAPTCHA alternatives, MFA options, password managers, and recovery flows. |
| INTERACT08 | Are disabled controls and unavailable states clear? | Explain why an action is unavailable and what user can do next. |
| INTERACT09 | What interaction test is required? | Test keyboard, screen reader, zoom, touch, reduced motion, and error recovery on key flows. |
| INTERACT10 | What interaction accessibility evidence should be kept? | Archive test matrix, issues, fixes, supported assistive tech, and residual limitations. |

| MEDIAA11Y07 | Are media controls accessible? | Ensure play, pause, volume, captions, full screen, speed, and transcript controls are reachable. |
| MEDIAA11Y08 | Are images or generated visuals safe and inclusive? | Check representation, stereotypes, sensitive depictions, cultural meaning, and inspection needs. |
| MEDIAA11Y09 | What media accessibility QA is required? | Review contrast, captions, transcript, audio description, controls, and visual alternatives. |

## Localization Scope

| Code | Access question | Workflow use |
|---|---|---|
| LOCSCOPE01 | What locales must be supported? | Define language, country, region, script, market, legal jurisdiction, and cultural audience. |
| LOCSCOPE02 | What content needs localization? | Include UI, docs, prompts, support, legal text, marketing, media, packaging, metadata, and release notes. |
| LOCSCOPE03 | What should remain untranslated? | Preserve brand names, product names, code, legal terms, units, identifiers, or regulated wording when needed. |
| LOCSCOPE04 | What locale-specific format matters? | Adapt dates, times, numbers, currency, addresses, phone, measurement, paper size, and sort order. |
| LOCSCOPE05 | What regional legal or policy variation matters? | Check privacy notices, claims, accessibility law, age rules, consumer rules, and product labeling. |
| LOCSCOPE06 | What language fallback applies? | Define default language, fallback locale, machine translation limit, and human review need. |
| LOCSCOPE07 | What localization asset source is authoritative? | Use glossary, translation memory, style guide, termbase, brand guide, or legal-approved copy. |
| LOCSCOPE08 | What expansion risk exists across locales? | Account for text length, layout, fonts, right-to-left scripts, input methods, and support capacity. |
| LOCSCOPE09 | What localization scope evidence is needed? | Keep locale list, content inventory, glossary, translation source, reviewer, and constraints. |
| LOCSCOPE10 | What localization scope artifact should be produced? | Produce localization brief with locales, content, formats, fallback, reviewers, and release constraints. |

## Translation And Terminology

| Code | Access question | Workflow use |
|---|---|---|
| TRANSTERM01 | What translation quality level is required? | Choose raw machine translation, post-editing, professional translation, transcreation, or legal-certified translation. |
| TRANSTERM02 | What terms must be consistent? | Maintain product names, domain terms, UI labels, legal terms, safety language, and workflow codes. |
| TRANSTERM03 | What terminology should be avoided? | Avoid offensive, outdated, ambiguous, regionally wrong, or legally risky terms. |
| TRANSTERM04 | What tone and register should translation use? | Define formal, informal, technical, instructional, marketing, support, or legal tone per locale. |
| TRANSTERM05 | What source ambiguity blocks translation? | Resolve unclear pronouns, missing context, abbreviations, UI string fragments, and culturally specific references. |
| TRANSTERM06 | What translation memory or glossary should be used? | Reuse approved translations and record new approved terms for future consistency. |
| TRANSTERM07 | What human review is required? | Use native reviewer, domain expert, legal reviewer, accessibility reviewer, or in-market reviewer. |
| TRANSTERM08 | What translation QA should run? | Check completeness, terminology, numbers, placeholders, tags, length, grammar, and context fit. |
| TRANSTERM09 | What translation issue should block release? | Block mistranslated legal/safety text, broken placeholders, offensive wording, or unreviewed high-risk content. |
| TRANSTERM10 | What translation artifact should be preserved? | Keep source strings, translations, glossary, reviewer notes, QA result, and approved version. |

| CULTFIT08 | What local exception should be documented? | Record locale-specific wording, feature difference, legal note, support exception, or content exclusion. |
| CULTFIT09 | What cultural-fit QA is required? | Review examples, imagery, claims, tone, symbols, and local expectations before release. |
| CULTFIT10 | What cultural-fit artifact should be archived? | Keep reviewer notes, localized decisions, exceptions, and approved local variants. |

## Device Network Constraints

| Code | Access question | Workflow use |
|---|---|---|
| DEVNET01 | What devices must be supported? | Define desktop, mobile, tablet, kiosk, assistive tech, print, low-end device, or offline context. |
| DEVNET02 | What network conditions matter? | Support low bandwidth, high latency, offline, intermittent, firewalled, or metered connections. |
| DEVNET03 | What platform constraints matter? | Check browser, OS, app version, file reader, screen size, input method, and enterprise restrictions. |
| DEVNET04 | What performance affects access? | Watch load time, file size, media bitrate, script weight, image resolution, and rendering time. |
| DEVNET05 | What offline or low-data alternative is needed? | Provide downloadable, printable, compressed, text-only, cached, or asynchronous path. |
| DEVNET06 | What device-specific layout risk exists? | Check reflow, truncation, orientation, zoom, long words, RTL, and touch target fit. |
| DEVNET07 | What unsupported environment should be disclosed? | State minimum requirements, limitations, and fallback support. |
| DEVNET08 | What device/network test matrix is needed? | Test key flows across target devices, browsers, networks, screen sizes, and file readers. |
| DEVNET09 | What device/network issue should block release? | Block inaccessible core flow, unreadable content, broken download, unusable mobile view, or no fallback. |
| DEVNET10 | What device/network evidence should be kept? | Archive test matrix, screenshots, performance results, known limits, and fallback instructions. |

## Usability Inclusion Testing

| Code | Access question | Workflow use |
|---|---|---|
| UINCLTEST01 | What representative users should test? | Include users across ability, language, device, role, experience level, and context where feasible. |
| UINCLTEST02 | What task should testing cover? | Test the real critical path: understand, decide, complete, recover, ask for help, and give feedback. |
| UINCLTEST03 | What accessibility manual test is needed? | Test keyboard, screen reader, zoom, captions, color contrast, forms, and error handling. |
| UINCLTEST04 | What localization test is needed? | Test translated UI/content in context, locale formats, layout expansion, and regional claims. |
| UINCLTEST05 | What comprehension test is needed? | Check whether target users understand instructions, labels, warnings, and next actions. |
| UINCLTEST06 | What inclusion feedback should be captured? | Capture confusion, exclusion, stereotypes, tone issues, support needs, and workaround requests. |
| UINCLTEST07 | What issue severity model applies? | Classify blocker, major, moderate, minor, cosmetic, or enhancement based on user impact. |
| UINCLTEST08 | What test result should block release? | Block when target users cannot complete core tasks, understand critical info, or access support. |
| UINCLTEST09 | What test follow-up is required? | Convert findings into fixes, accepted risks, accommodations, documentation, or backlog items. |
| UINCLTEST10 | What usability/inclusion evidence should be preserved? | Keep participants, tasks, findings, severity, decisions, fixes, and residual risks. |

## Accommodation And Support

| Code | Access question | Workflow use |
|---|---|---|
| ACCOMSUP01 | What alternative format should be offered? | Provide text, audio, captioned video, large print, accessible PDF, CSV, HTML, or simplified guide. |
| ACCOMSUP02 | What human support path is needed? | Provide contact, live help, callback, interpreter, escalation, or assisted completion. |
| ACCOMSUP03 | What language support is needed? | Offer translated support, interpreter path, bilingual documentation, or locale-specific help. |
| ACCOMSUP04 | What disability accommodation process applies? | Define request path, response time, documentation need, privacy, and responsible team. |
| ACCOMSUP05 | What fallback should be available when automation fails? | Provide manual review, alternate channel, offline submission, or assisted workflow. |
| ACCOMSUP06 | What support staff training is required? | Train on respectful language, accessibility features, accommodations, localization, and escalation. |
| ACCOMSUP07 | What support scripts should avoid? | Avoid blaming users, exposing private details, making legal/medical promises, or denying accommodations prematurely. |
| ACCOMSUP08 | What support metric should be monitored? | Track accommodation requests, completion rate, response time, language issues, and accessibility complaints. |
| ACCOMSUP09 | What support issue should trigger workflow change? | Update workflow when repeated barriers, support escalations, complaints, or manual workarounds appear. |
| ACCOMSUP10 | What accommodation/support artifact should be kept? | Archive support paths, scripts, training, request handling, metrics, and improvement actions. |

## Accessibility Localization Audit

| Code | Access question | Workflow use |
|---|---|---|
| A11YLOCAUD01 | Does the output meet declared accessibility standards? | Verify against chosen standard, policy, or acceptance checklist. |
| A11YLOCAUD02 | Does the output meet declared localization scope? | Confirm every required locale, format, term, and regional variant is complete. |
| A11YLOCAUD03 | Are core tasks usable by target users? | Verify completion, comprehension, recovery, and support across representative contexts. |
| A11YLOCAUD04 | Are accessibility and localization issues tracked? | Ensure issues have severity, owner, fix, acceptance, or documented residual risk. |
| A11YLOCAUD05 | Are artifacts accessible after final export or release? | Recheck exported PDFs, videos, dashboards, packages, documents, and published pages. |
| A11YLOCAUD06 | Are translations and cultural adaptations approved? | Verify reviewer signoff, legal-sensitive text, glossary, and in-context QA. |
| A11YLOCAUD07 | Are accommodations and fallback paths documented? | Confirm alternate formats, support path, language help, and manual fallback exist. |
| A11YLOCAUD08 | Are known limitations disclosed appropriately? | State unsupported devices, languages, assistive tech, formats, or access constraints. |
| A11YLOCAUD09 | What accessibility/localization evidence should be archived? | Keep checklists, scans, manual tests, screenshots, reviewer notes, and approvals. |
| A11YLOCAUD10 | What improvement should feed the next iteration? | Convert audit gaps into backlog, template updates, standards, training, or monitoring. |
