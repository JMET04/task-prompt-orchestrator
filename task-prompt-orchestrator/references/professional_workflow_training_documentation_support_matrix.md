# Professional Workflow Training Documentation Support Matrix

Use this matrix when a workflow must teach users, document procedures, transfer knowledge, onboard teams, support operations, or measure whether people can actually use the workflow.

## Training And Support Scope

| Code | Workflow question | Workflow use |
|---|---|---|
| TRNSCOPE01 | Who must learn or operate this workflow? | Identify learners, operators, reviewers, support staff, admins, customers, and secondary audiences. |
| TRNSCOPE02 | What capability must each audience gain? | Convert vague training needs into concrete job tasks, decisions, and quality behaviors. |
| TRNSCOPE03 | What is the current support burden? | Find repeated questions, defects, escalations, handoff misses, and adoption friction. |
| TRNSCOPE04 | What must be self-serve versus instructor-led? | Split documentation, tutorials, live sessions, coaching, and escalation support. |
| TRNSCOPE05 | What knowledge is stable versus changing? | Separate durable concepts from volatile steps, tool screens, policies, or version-specific details. |
| TRNSCOPE06 | What regulatory or safety training applies? | Add required attestations, role restrictions, audit evidence, and refresher cadence. |
| TRNSCOPE07 | What languages, locales, and accessibility needs apply? | Plan inclusive delivery for literacy, device, assistive technology, language, and region differences. |
| TRNSCOPE08 | What support channels will exist? | Define help center, chat, email, office hours, ticket queue, escalation, and community channels. |
| TRNSCOPE09 | What must be ready before rollout? | Create readiness gates for docs, training assets, exercises, support staffing, and feedback loops. |
| TRNSCOPE10 | What evidence proves enablement is complete? | Define completion, proficiency, adoption, satisfaction, support deflection, and quality metrics. |

## Audience Capability Analysis

| Code | Workflow question | Workflow use |
|---|---|---|
| AUDCAP01 | What roles use the workflow differently? | Build role-specific learning paths instead of one generic tutorial. |
| AUDCAP02 | What prior knowledge can be assumed? | Tune explanations, prerequisites, glossary depth, and practice difficulty. |
| AUDCAP03 | What skill gaps block success? | Target training at the real failure points rather than covering every feature. |
| AUDCAP04 | What misconceptions are likely? | Add corrective examples, anti-patterns, and checks for common wrong mental models. |
| AUDCAP05 | What motivation or resistance exists? | Address perceived value, effort, risk, incentives, and change fatigue. |
| AUDCAP06 | What environmental constraints affect learning? | Account for time, bandwidth, device, schedule, permissions, and manager support. |
| AUDCAP07 | What expert users can become champions? | Identify peer coaches, office-hour helpers, reviewers, and feedback sources. |
| AUDCAP08 | What novice path prevents overload? | Stage learning into first success, daily use, advanced use, and troubleshooting. |
| AUDCAP09 | What advanced path prevents stagnation? | Add power-user modules, optimization guides, automation, and contribution paths. |
| AUDCAP10 | What support segmentation is needed? | Route beginner, intermediate, expert, admin, and edge-case users to different support assets. |

## Learning Objectives

| Code | Workflow question | Workflow use |
|---|---|---|
| LEARNOBJ01 | What observable task should the learner perform? | Write objectives as actions that can be demonstrated and assessed. |
| LEARNOBJ02 | What decision should the learner make correctly? | Include judgment rules, tradeoffs, thresholds, and escalation criteria. |
| LEARNOBJ03 | What quality standard should the learner meet? | Tie training to acceptance criteria, examples, rubrics, and validation checks. |
| LEARNOBJ04 | What mistakes should the learner avoid? | Convert defects and support tickets into learning objectives. |
| LEARNOBJ05 | What vocabulary must be shared? | Define terms, acronyms, artifact names, states, roles, and handoff labels. |
| LEARNOBJ06 | What policy or safety rule must be remembered? | Add mandatory controls, privacy limits, approval gates, and prohibited actions. |
| LEARNOBJ07 | What sequence must be followed? | Teach state transitions, dependencies, checklists, and phase gates. |
| LEARNOBJ08 | What tool operation must be practiced? | Include clicks, commands, prompts, file paths, forms, integrations, and validation outputs. |
| LEARNOBJ09 | What transfer goal matters after training? | Ensure learners can apply the workflow to new cases, not only repeat the demo. |
| LEARNOBJ10 | What assessment proves the objective? | Map each objective to quiz, lab, review, simulation, certification, or manager signoff. |

## Documentation System

| Code | Workflow question | Workflow use |
|---|---|---|
| DOCSYS01 | What documentation types are required? | Choose quickstart, SOP, reference, FAQ, runbook, tutorial, checklist, release note, or policy. |
| DOCSYS02 | Where should documentation live? | Select repository, knowledge base, help center, wiki, LMS, shared drive, or embedded product help. |
| DOCSYS03 | What structure makes docs findable? | Use names, tags, index pages, search keywords, roles, lifecycle stage, and task labels. |
| DOCSYS04 | What source of truth controls content? | Define owner, version, review cadence, approval, archive, and change log. |
| DOCSYS05 | What format best fits the task? | Pick text, screenshots, video, diagrams, tables, templates, examples, or interactive walkthroughs. |
| DOCSYS06 | What must be copyable or reusable? | Provide prompt packets, commands, forms, checklists, sample files, and response templates. |
| DOCSYS07 | What must be localized or adapted? | Plan translation, regional terms, screenshots, examples, legal disclaimers, and reading level. |
| DOCSYS08 | What accessibility requirements apply to docs? | Add headings, alt text, captions, contrast, keyboard navigation, plain language, and export checks. |
| DOCSYS09 | How will stale docs be detected? | Add owner review, broken-link checks, tool-version checks, ticket feedback, and usage analytics. |
| DOCSYS10 | What doc evidence is needed for audit? | Preserve published version, approver, effective date, training mapping, and change history. |

## Onboarding Path

| Code | Workflow question | Workflow use |
|---|---|---|
| ONBOARD01 | What is the first successful outcome? | Design onboarding around a small useful win rather than full mastery. |
| ONBOARD02 | What account, access, or setup is required? | Include prerequisites, permissions, sample data, environment checks, and setup validation. |
| ONBOARD03 | What orientation does the user need? | Explain purpose, roles, workflow map, artifacts, support channels, and boundaries. |
| ONBOARD04 | What guided path should a new user follow? | Sequence welcome, setup, demo, practice, first live use, review, and support follow-up. |
| ONBOARD05 | What should be skipped for experienced users? | Provide fast-track onboarding, migration notes, and advanced entry points. |
| ONBOARD06 | What handoff happens from trainer to operator? | Define readiness evidence, owner transfer, support contact, and next check-in. |
| ONBOARD07 | What onboarding automation helps? | Use checklists, emails, in-app tours, LMS assignments, templates, and scheduled reminders. |
| ONBOARD08 | What early-warning signs show onboarding failed? | Track inactivity, errors, repeated questions, support tickets, and missed milestones. |
| ONBOARD09 | What manager or sponsor role is needed? | Give managers coaching points, progress visibility, and reinforcement tasks. |
| ONBOARD10 | What onboarding closeout record is required? | Capture completion, access status, proficiency evidence, open questions, and next path. |

## Training Path

| Code | Workflow question | Workflow use |
|---|---|---|
| TRAINPATH01 | What modules should the curriculum include? | Build modular learning around tasks, roles, risks, tools, and lifecycle phases. |
| TRAINPATH02 | What order should modules follow? | Sequence from concepts to demo, guided practice, independent work, review, and mastery. |
| TRAINPATH03 | What delivery mode fits each module? | Choose self-paced docs, video, workshop, lab, coaching, cohort, or embedded guidance. |
| TRAINPATH04 | What examples should be used? | Use realistic cases, edge cases, bad examples, before-after examples, and domain-specific scenarios. |
| TRAINPATH05 | What timebox keeps training usable? | Set duration, prerequisites, breaks, practice time, and follow-up assignments. |
| TRAINPATH06 | What materials does the trainer need? | Prepare agenda, script, slides, demo data, exercises, answer key, and facilitation notes. |
| TRAINPATH07 | What materials does the learner need? | Provide workbook, checklist, sample inputs, tool access, glossary, and reference links. |
| TRAINPATH08 | What reinforcement happens after training? | Schedule nudges, office hours, peer review, practice assignments, and performance feedback. |
| TRAINPATH09 | How will training adapt by cohort? | Tailor examples, pacing, language, support depth, and certification to each learner group. |
| TRAINPATH10 | What training version must be maintained? | Link curriculum to workflow version, release date, owner, and retired modules. |

## Practice Simulation Labs

| Code | Workflow question | Workflow use |
|---|---|---|
| TRAINPRACTICE01 | What realistic practice case is needed? | Create scenarios that mirror actual inputs, constraints, tools, and quality expectations. |
| TRAINPRACTICE02 | What safe sandbox should be used? | Avoid live customer data, production systems, irreversible actions, or confidential materials. |
| TRAINPRACTICE03 | What step-by-step lab guides practice? | Provide setup, task, hints, expected output, validation, and reflection prompts. |
| TRAINPRACTICE04 | What challenge tests transfer? | Add a novel case that requires judgment rather than copying the demo. |
| TRAINPRACTICE05 | What common failure should be simulated? | Practice missing inputs, bad sources, tool failure, ambiguous requests, and escalation. |
| TRAINPRACTICE06 | What peer or reviewer feedback is needed? | Use rubrics, pair review, trainer review, or automated checks on practice outputs. |
| TRAINPRACTICE07 | What answer key or exemplar is needed? | Provide benchmark outputs, reasoning notes, and acceptable variation. |
| TRAINPRACTICE08 | What practice data must be refreshed? | Keep examples current, safe, localized, and representative of real work. |
| TRAINPRACTICE09 | What practice evidence should be captured? | Save completion status, scores, attempts, feedback, and readiness recommendations. |
| TRAINPRACTICE10 | What live transition follows practice? | Define supervised first use, shadowing, review gate, or gradual permission increase. |

## Support Content And FAQ

| Code | Workflow question | Workflow use |
|---|---|---|
| SUPPORTC01 | What questions are most likely? | Build FAQ entries from onboarding issues, ticket history, user interviews, and pilot feedback. |
| SUPPORTC02 | What answer style is needed? | Use short direct answers, decision trees, examples, screenshots, or links to full docs. |
| SUPPORTC03 | What support content prevents tickets? | Create quick fixes, setup checks, glossary, known issues, and self-diagnosis guides. |
| SUPPORTC04 | What content supports frontline agents? | Provide macros, triage scripts, escalation rules, and verified knowledge snippets. |
| SUPPORTC05 | What content belongs inside the workflow? | Embed tooltips, inline help, checklist notes, validation messages, and template instructions. |
| SUPPORTC06 | What content must be approved? | Route legal, policy, safety, brand, or regulated answers through proper review. |
| SUPPORTC07 | What content must be personalized? | Adapt support by role, plan, region, language, feature flag, or maturity level. |
| SUPPORTC08 | How will FAQ gaps be found? | Review search terms, unresolved tickets, repeated chat questions, and failed help journeys. |
| SUPPORTC09 | What support handoff points exist? | Link self-service, human support, specialist escalation, engineering, vendor, and incident routes. |
| SUPPORTC10 | What support content metrics matter? | Track deflection, helpfulness, search success, time to answer, reopen rate, and outdated content. |

## Troubleshooting Runbooks

| Code | Workflow question | Workflow use |
|---|---|---|
| TROUBLE01 | What symptoms indicate a problem? | List visible errors, user reports, missing outputs, quality failures, and monitoring signals. |
| TROUBLE02 | What first checks should be run? | Define permissions, inputs, versions, connectivity, logs, config, and known issue checks. |
| TROUBLE03 | What decision tree isolates root cause? | Route by symptom, scope, severity, affected users, reproducibility, and recent changes. |
| TROUBLE04 | What safe repair actions are allowed? | Provide reversible steps, retries, cache clears, config fixes, rollback, or escalation. |
| TROUBLE05 | What actions are prohibited? | Identify destructive, insecure, noncompliant, or unapproved repair attempts. |
| TROUBLE06 | What evidence should support collect? | Capture screenshots, logs, input IDs, timestamps, user role, environment, and reproduction steps. |
| TROUBLE07 | What escalation criteria apply? | Escalate by severity, data risk, customer impact, SLA breach, security, or repeated failure. |
| TROUBLE08 | What user communication is needed? | Provide acknowledgement, workaround, expected resolution, status updates, and closeout. |
| TROUBLE09 | What post-fix verification is required? | Confirm restored function, no regression, user acceptance, and ticket closure. |
| TROUBLE10 | What runbook improvement follows? | Update docs, FAQ, monitoring, training, and known-issue records after repeated incidents. |

| KTRANS07 | What continuity risk exists? | Plan backup owner, overlap, documentation gap closure, and emergency escalation. |
| KTRANS08 | What receiver confirmation is required? | Use checklist signoff, teach-back, first-run proof, or manager acceptance. |
| KTRANS09 | What remains unresolved after handoff? | Track open questions, debt, pending approvals, missing docs, and next review date. |
| KTRANS10 | What archive record should be kept? | Save final package, signoff, date, owner, version, and location for future retrieval. |

| ENABLEMET07 | What analytics source will be used? | Use LMS, product analytics, ticketing, docs analytics, surveys, QA reviews, or audit logs. |
| ENABLEMET08 | What review cadence improves enablement? | Schedule weekly launch review, monthly content review, quarterly curriculum refresh, or incident-triggered update. |
