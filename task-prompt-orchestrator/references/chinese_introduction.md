# Task Prompt Orchestrator 中文介绍

## 一句话定位

`task-prompt-orchestrator` 是一个面向 Codex 的任务提示词编排 skill。它不是单一领域的提示词模板，而是一个低 token 消耗的工作流入口：先理解用户目标和输入材料，再扫描当前可用的 skill、插件、工具、脚本和本地索引，选择最小但足够专业的工作流，最后把任务拆成可执行的提示词包并完成验证。

## 它解决什么问题

很多用户的真实请求不是标准提示词，而是混合了目标、文件、已有上下文、模糊约束和临时想法。直接把这类请求丢给模型，容易出现三个问题：

- 上来就写大段通用提示词，真正需要的输入反而没收齐。
- 明明本机已经有合适的 skill、插件、脚本或索引，却没有先调用现成能力。
- 为了找一个模板，把整套提示词库或大量参考文件读进上下文，造成 token 浪费和噪声。

这个 skill 的核心价值是把“我要做什么”变成“下一步该用哪个能力、读哪一个最小索引、执行哪几个步骤、用什么标准验收”。

## 核心功能

1. 任务接收与信息规整
   - 提取目标、原始请求、已提供文件/链接/截图/数据、当前工作区状态、输出目标、约束、验收标准和执行权限。
   - 只在缺失信息会实质改变结果时提问，避免用过多澄清打断执行。

2. 能力优先扫描
   - 在自造流程前，先检查当前可用的 skills、plugins/connectors、MCP/app tools、CLI 工具、本地脚本、已有索引和提示词库。
   - 当用户插件较少时，可按任务需要推荐或安装常用 GitHub、Codex、OpenAI、数据、设计、部署类插件。

3. 专业工作流路由
   - 根据任务类型选择对应工作流：image2、设计、研究、代码、文档、数据分析、营销、自动化、法律、金融、合规、产品、商业、教育、评审等。
   - 对复杂任务先拆分步骤，再为每一步选择合适提示词模板或专门能力。

4. 提示词优化与提示词包生成
   - 把用户原话压缩成清晰 brief。
   - 为每个步骤生成包含角色、目标、上下文、输入、流程、输出格式、约束、质量检查和交接信息的 prompt packet。

5. 执行与验收
   - 如果用户要求的是行动，就继续执行，而不是停留在计划或提示词优化。
   - 使用测试、文件检查、视觉检查、链接、计数、引用闭环或其他证据来确认结果。

## 实现原理

这个 skill 采用“渐进式披露”和“索引优先”的实现方式。

第一层是 metadata，也就是 `SKILL.md` 的 `name` 和 `description`。这部分始终很短，用来告诉 Codex 什么时候触发这个 skill。

第二层是 `SKILL.md` 主体。它只保留核心合同、输入信息规范、任务拆解方法、模板选择入口、执行和验收规则。正常任务触发后，Codex 主要读取这一层，不会默认加载完整提示词库。

第三层是 `references/` 里的专业索引和矩阵。只有当任务落到具体方向时，才读取相应文件。例如：

- 广泛 AI 工作流或新手控制场景，先读 `ai_full_domain_activation_directory.md`。
- 插件/skill 扫描和自动安装场景，读 capability/plugin 相关矩阵。
- image2 场景，先读 `D:\image2专业提示词库\indexes\IMAGE2_VERTICAL_PROMPT_INDEX.md`，再进入具体提示词文件。
- 包装 CAD、工程图、制造标注等场景，再读 `IMAGE2_PRODUCTION_SCENARIO_INDEX.md`。

第四层是 `scripts/`。像 `route_ai_workflow.py` 这样的脚本可以先做快速路由，把用户请求映射到少量候选工作流；`validate_activation_surface.py` 用于维护低成本入口、路由锚点和 starter pack 的一致性。脚本能减少每次重复读取和重写大段规则的成本。

## 低 token 消耗设计

这个 skill 明确强调低 token 消耗。它不会每次都读取完整提示词库，也不会把所有专业矩阵一次性塞进上下文。

它的默认策略是：

1. 先读最小入口
   - 先用 `SKILL.md` 和必要的总索引判断方向。

2. 再读最小匹配文件
   - 只读取当前任务需要的一个或少数几个 reference。

3. 优先用索引定位
   - 对 image2 等大提示词库，先看 index，再打开命中的具体 prompt 文件。

4. 用脚本做快速筛选
   - 能用 `route_ai_workflow.py` 初筛时，先运行脚本得到候选路线，再决定是否读取详细说明。

5. 不把介绍当执行上下文
   - 本中文介绍只在用户询问功能、原理、发布说明或中文文档时读取；普通任务执行不需要加载它。

因此，它更像一个轻量调度器，而不是一个每次都全量展开的超大提示词合集。

## 典型使用方式

用户可以这样调用：

```text
使用 task-prompt-orchestrator，先收集我的目标、输入材料、约束、可用插件和验收标准，再选择最小可用工作流并执行。
```

也可以用于更具体的场景：

```text
帮我扫描当前可用的插件和 skill，判断这个任务应该调用哪些能力，并生成执行工作流。
```

```text
我要做 image2 包装 CAD 图纸提示词，先根据索引找到最专业的方向，不要读取整个提示词库。
```

## 不适合做什么

这个 skill 不应该替代已经更专业的领域 skill。比如任务明确是 GitHub PR 审查、Word 文档处理、数据仪表盘、前端应用构建或 image2 具体出图，它应该先路由到最合适的专业能力，再负责输入整理、提示词包生成、执行协调和验收。

## 对用户的承诺

- 先找现成能力，再设计新流程。
- 先问真正阻塞的问题，不为完美信息反复打断。
- 先用索引和脚本缩小范围，不全量读取提示词库。
- 先拆成可执行步骤，再写提示词。
- 先验证结果，再宣布完成。
